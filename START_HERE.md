# С чего начать — пошаговая инструкция

Этот документ ведёт от нулевого состояния до работающего бота. Идите по шагам сверху вниз, не пропускайте.

---

## День 1: подготовка (1–2 часа)

### Шаг 1. Создайте бота в Telegram

1. Откройте Telegram, найдите `@BotFather`
2. Отправьте `/newbot`
3. Введите имя бота: `Медбот для клиник`
4. Введите username: должен заканчиваться на `bot`, например `myclinic_supplies_bot`
5. Скопируйте токен — выглядит как `1234567890:ABCdefGhIJK_lmnopqrstuvwxyz`

Сохраните токен — он нужен на шаге 5.

### Шаг 2. Узнайте свой Telegram ID

1. Найдите `@userinfobot`
2. Отправьте `/start`
3. Бот пришлёт ваш ID — число вроде `123456789`

Сохраните — нужен для роли админа.

### Шаг 3. Установите Docker на компьютер

**Windows / macOS:** скачайте Docker Desktop с docker.com и установите.

**Linux:**
```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# перезайдите в систему
```

Проверка:
```bash
docker --version
docker compose version
```

### Шаг 4. Распакуйте проект

```bash
tar -xzf medbot.tar.gz
cd medbot
```

### Шаг 5. Создайте файл .env

```bash
cp .env.example .env
```

Откройте `.env` любым текстовым редактором. Замените:

```ini
BOT_TOKEN=ваш_токен_от_BotFather
ADMIN_TELEGRAM_IDS=ваш_telegram_id
```

Остальные параметры пока не трогайте — они уже настроены для работы с мок-сервером.

## День 1: первый запуск с мок-сервером (30 минут)

### Шаг 6. Запустите проект локально

```bash
docker compose --profile dev up -d --build
```

Флаг `--profile dev` поднимает дополнительный сервис `mock_1c` — это заглушка вместо настоящей 1С. С ней можно тестировать бота не дожидаясь готовности 1С-программиста.

Проверьте что всё запустилось:
```bash
docker compose ps
```

Должно быть 4 сервиса в статусе `running`: `postgres`, `redis`, `mock_1c`, `bot`.

### Шаг 7. Посмотрите логи

```bash
docker compose logs -f bot
```

Через 10–20 секунд увидите примерно:
```
[INFO] Bot started: @myclinic_supplies_bot
[INFO] [sync products] {'created': 10, 'updated': 0, 'deactivated': 0}
[INFO] [sync stock] {'updated': 0}
```

Это означает: бот запустился, синхронизация с моком работает, в БД появилось 10 тестовых товаров.

Если видите ошибку — обратите внимание на её текст. Самые частые:
- `Connection refused` — Postgres ещё не успел запуститься, подождите 10 секунд
- `Unauthorized` — неправильный BOT_TOKEN, перепроверьте `.env`

### Шаг 8. Загрузите начальные лимиты в БД

В отдельном терминале:
```bash
docker compose exec postgres psql -U medbot -d medbot -f /docker-entrypoint-initdb.d/initial_limits.sql
```

Если выдаёт «file not found» — выполните вручную:
```bash
docker compose exec -T postgres psql -U medbot -d medbot < sql/initial_limits.sql
```

### Шаг 9. Откройте бота в Telegram

1. Найдите вашего бота по username (тот что задавали в @BotFather)
2. Отправьте `/start`
3. Бот попросит ИНН — введите тестовый: `300123456`
4. Бот скажет «Найдена организация: Городская клиническая больница №1, уровень VIP»
5. Подтвердите регистрацию
6. Появится главное меню

Поздравляю — вы только что прошли регистрацию по ИНН через мок-сервер 1С!

### Шаг 10. Сделайте тестовый заказ

1. Нажмите «🛍 Каталог»
2. Выберите категорию «Расходные материалы» → «Перчатки и маски»
3. Выберите «Перчатки латексные...»
4. Нажмите «🛒 Добавить 1 шт»
5. Вернитесь в меню → «🛒 Корзина»
6. Нажмите «✅ Оформить заказ»
7. Введите комментарий (или просто `-`)
8. Бот пришлёт PDF счёта и PDF договора (тестовые)

### Шаг 11. Эмулируйте оплату

В терминале посмотрите ID заказа:
```bash
curl http://localhost:8001/_admin/orders
```

Скопируйте `one_c_id` из ответа и пометьте оплаченным:
```bash
curl -X POST http://localhost:8001/_admin/orders/<вставьте_one_c_id>/mark-paid
```

В течение 10 минут (можно сократить интервал в `.env`, см. ниже) бот пришлёт уведомление: «✅ Оплата получена».

### Если хотите ускорить тестирование

В `.env` поменяйте интервалы на 30 секунд:
```ini
SYNC_PAYMENTS_INTERVAL=30
SYNC_ORDERS_INTERVAL=30
```

Перезапустите бота:
```bash
docker compose restart bot
```

Теперь после `mark-paid` уведомление приходит почти сразу.

## Дни 2–3: организационные задачи (параллельно)

Не ждите окончания этого этапа чтобы продолжать тесты — мок-сервер позволит работать с ботом дальше. Эти задачи нужно запустить сейчас, потому что они занимают недели.

### Шаг 12. Свяжитесь с 1С-программистом

Отправьте ему письмо или сообщение примерно такого содержания:

> Делаем Telegram-бот для клиентов. Бот должен по HTTP получать данные из 1С: проверять контрагентов по ИНН, выгружать каталог номенклатуры, создавать заказы со счётами и проверять оплату.
>
> Прилагаю готовый шаблон HTTP-сервиса на встроенном языке (`onec_examples/HTTPService.bsl`). Нужно адаптировать под нашу конфигурацию.
>
> Когда сможем встретиться обсудить детали? Особенно интересует: есть ли реквизит «УровеньКлиента» у Контрагентов, есть ли печатные формы счёта и договора, можно ли автоматически генерировать PDF.

Передайте ему файлы:
- `onec_examples/HTTPService.bsl` — пример кода
- `mock_1c/server.py` — спецификация эндпоинтов (показывает какой JSON ожидает бот)

### Шаг 13. Согласуйте уровни клиентов с менеджерами

С отделом продаж определите:

1. Сколько уровней клиентов будет (стандартно три: VIP / Стандартный / Ограниченный, но можно больше)
2. Какие лимиты на каждый уровень — отредактируйте `sql/initial_limits.sql` под реальные цифры
3. Нужны ли особые правила для отдельных категорий товаров (например, дорогое оборудование только для VIP)

### Шаг 14. Получите тестовую базу 1С

От вашего 1С-программиста или сисадмина:
- IP/домен сервера 1С
- Имя базы
- Логин/пароль для интеграционного пользователя
- Минимум 5–10 тестовых контрагентов с разными уровнями
- Минимум 50–100 товаров в каталоге для проверки

## Неделя 2: переход с мок-сервера на реальную 1С

После того как 1С-программист опубликует HTTP-сервис.

### Шаг 15. Проверьте эндпоинты 1С через curl

Перед тем как переключать бота — убедитесь что 1С отвечает корректно:

```bash
# Проверка ИНН
curl -u integration_bot:password \
  "http://your-1c-server/YourBase/hs/bot/clients/check?inn=300123456"

# Должно вернуть JSON с found, name, tier, etc.
```

```bash
# Проверка каталога
curl -u integration_bot:password \
  "http://your-1c-server/YourBase/hs/bot/catalog/products" | head -50

# Проверьте кодировку — кириллица должна быть читаемой
```

Если что-то не так — сначала просите 1С-программиста починить, а уже потом подключайте бота.

### Шаг 16. Переключите бота на реальную 1С

В `.env` замените:

```ini
ONE_C_URL=http://your-1c-server.uz/YourBase
ONE_C_USER=integration_bot
ONE_C_PASSWORD=ваш_реальный_пароль
```

Перезапустите:
```bash
docker compose down
docker compose up -d --build
```

(Без `--profile dev` — мок больше не нужен.)

### Шаг 17. Первая синхронизация с реальной 1С

Посмотрите логи:
```bash
docker compose logs -f bot
```

Через 15 минут (или сразу — если уменьшили интервал) увидите:
```
[INFO] [sync products] {'created': 700, 'updated': 0, 'deactivated': 0}
```

Если ошибки — проверяйте:
1. Доступен ли HTTP-сервис 1С с того же сервера где бот? (`curl` изнутри контейнера)
2. Кодировка — UTF-8 или Windows-1251?
3. Правильно ли называются поля JSON? (Должны совпадать с теми что ждёт `services/onec_client.py`)

### Шаг 18. Проверьте реальный заказ

1. Зарегистрируйтесь в боте под своим ИНН (должен быть реально в 1С)
2. Сделайте тестовый заказ на минимальную сумму
3. Проверьте в 1С: появился ли документ ЗаказПокупателя с теми же товарами
4. Проверьте: пришёл ли в Telegram настоящий PDF счёта (не тестовый)
5. Проведите оплату в 1С — посмотрите придёт ли уведомление в бот

Это главная веха — после неё проект «реален».

## Неделя 3: развёртывание на VPS

Когда локальное тестирование с реальной 1С прошло успешно.

### Шаг 19. Арендуйте VPS

Рекомендую:
- **Hetzner CX22** (~5€/мес): 2 vCPU, 4 GB RAM. Берите регион Falkenstein или Helsinki.
- **Contabo VPS S** (~7€/мес): 4 vCPU, 8 GB RAM, NVMe.

Установите Ubuntu 22.04 LTS.

### Шаг 20. Настройте VPS

```bash
ssh root@your-vps-ip

# Обновление
apt update && apt upgrade -y

# Создаём пользователя
adduser medbot
usermod -aG sudo medbot

# Docker
curl -fsSL https://get.docker.com | sh
usermod -aG docker medbot

# Файрвол
ufw allow OpenSSH
ufw enable

# Перезайдите как medbot
su - medbot
```

### Шаг 21. Перенесите проект

С локальной машины:
```bash
scp medbot.tar.gz medbot@your-vps-ip:~/
```

На VPS:
```bash
tar -xzf medbot.tar.gz
cd medbot
cp .env.example .env
nano .env   # тот же .env что и локально, но с production-настройками
```

Важно: `POSTGRES_PASSWORD` сделайте сильным, а не дефолтным `medbot`.

### Шаг 22. Запустите в production

```bash
docker compose up -d --build
docker compose logs -f bot
```

Загрузите лимиты:
```bash
docker compose exec -T postgres psql -U medbot -d medbot < sql/initial_limits.sql
```

### Шаг 23. Настройте автобэкапы

Создайте `~/backup.sh`:
```bash
#!/bin/bash
TIMESTAMP=$(date +%Y%m%d-%H%M)
docker compose -f /home/medbot/medbot/docker-compose.yml exec -T postgres \
    pg_dump -U medbot medbot | gzip > /home/medbot/backups/db-$TIMESTAMP.sql.gz
# Удаляем бэкапы старше 30 дней
find /home/medbot/backups -name "db-*.sql.gz" -mtime +30 -delete
```

```bash
chmod +x ~/backup.sh
mkdir -p ~/backups
crontab -e
# Добавьте строку — бэкап каждые 6 часов:
0 */6 * * * /home/medbot/backup.sh
```

### Шаг 24. Подключите Sentry для мониторинга ошибок

1. Зарегистрируйтесь на sentry.io (бесплатный план хватит)
2. Создайте проект Python
3. Скопируйте DSN

Добавьте в `requirements.txt`:
```
sentry-sdk==2.13.0
```

В начале `main.py`:
```python
import sentry_sdk
import os

if dsn := os.getenv("SENTRY_DSN"):
    sentry_sdk.init(dsn=dsn, traces_sample_rate=0.1)
```

В `.env`:
```ini
SENTRY_DSN=https://...@...sentry.io/...
```

Пересоберите и перезапустите.

## Неделя 4: пилотный клиент

### Шаг 25. Найдите 1–2 лояльных клиентов

Не больше! С двумя клиентами проще ловить и чинить баги.

Договоритесь:
- Они будут писать обо всех проблемах в личку
- Первые 1–2 недели возможны сбои
- За это им — особые условия (скидка, приоритет, etc.)

Отправьте им ссылку на бота и инструкцию из 3 пунктов:
1. Откройте бота
2. Отправьте `/start`
3. Введите ИНН вашей организации

Попросите оформить настоящий заказ на небольшую сумму. Сопровождайте процесс лично.

### Шаг 26. Соберите обратную связь

После каждого заказа в первые 2 недели — звоните клиенту и спрашивайте:
- Что было непонятно?
- Что неудобно?
- Чего не хватало?

Это самые важные данные за весь проект. Лучшая обратная связь приходит от реальных пользователей.

## Что делать если что-то сломалось

### Бот не отвечает

```bash
docker compose ps                    # все сервисы запущены?
docker compose logs --tail=100 bot   # что в логах?
```

Чаще всего: упал контейнер, нужно перезапустить:
```bash
docker compose restart bot
```

### Каталог пустой

```bash
docker compose exec -T postgres psql -U medbot -d medbot \
  -c "SELECT entity, status, details FROM sync_logs ORDER BY started_at DESC LIMIT 5;"
```

Посмотрите последние записи синхронизации. Если все `error` — проверяйте связь с 1С.

### 1С недоступна — заказы зависают

Это нормально — заказы попадают в Redis-очередь и отправятся когда связь восстановится. Если 1С простаивает дольше часа, вручную проверьте:

```bash
# Запросы в очереди
docker compose exec redis redis-cli LRANGE orders:pending_1c 0 -1
```

### Клиент жалуется что не может оформить заказ

Проверьте лимиты для его уровня:
```sql
SELECT * FROM tier_limits
WHERE tier = (SELECT client_tier FROM organizations WHERE inn = '<его ИНН>');
```

Если лимит занижен — поднимите его в `tier_limits`.
